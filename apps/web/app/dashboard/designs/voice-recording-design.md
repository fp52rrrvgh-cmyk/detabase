# 語音記帳設計

## 定位

讓小新在 Telegram 用自然語言一句話完成記帳，支援：
- 純語音訊息（開車時）
- 打字（方便時）
- 從 Hermes 小馬對話直接呼叫

不需要開啟任何 app 或網頁。

## 輸入格式

```
[金額] [備註/商店] [分類] [帳戶]
```

### 範例

```
加油 5000 現金
→ 金額：5000，備註：加油，分類：自動判斷，帳戶：現金

午餐 120
→ 金額：120，備註：午餐，分類：餐飲，帳戶：預設

買了輪胎 18000 從玉山
→ 金額：18000，備註：買了輪胎，分類：自動判斷，帳戶：玉山

全聯 350 生活
→ 金額：350，備註：全聯，分類：生活日用，帳戶：預設

收入 39760 月薪 玉山
→ 收入 39760，備註：月薪，分類：薪資，帳戶：玉山
```

### 判斷邏輯

1. 若訊息以「收入」「薪資」開頭 → 記為 income
2. 否則預設 expense
3. 金額：第一個數字，支援逗號（1,500）和小數（23.45）
4. 分類：找關鍵字對應分類名稱，找不到就留空讓系統自動分類
5. 帳戶：找關鍵字對應帳戶名稱，找不到用預設

## 實作方式

### 選項 A — 僅透過 Hermes 處理（推薦先做）

小馬在 Telegram 對話中收到使用者的記帳訊息，自己解析並呼叫 Edge Function 寫入資料庫。

流程：
```
使用者 → Telegram: "加油 5000 現金"
小馬解析 → 呼叫 log-finance-activity Edge Function
回覆: "✅ 已記錄：加油 TWD 5,000（交通）從 現金"
```

**不需要額外 bot、不需要 webhook、不需要新 infrastructure。**

### 選項 B — 專屬 Telegram Bot（後續擴充）

如果以後要支援語音轉文字 + 獨立 bot 記帳，可以開第二個 bot token 專門處理記帳。

## Edge Function 需要調整

## Edge Function

目前的 `log-finance-activity` 需要接受從 Hermes 端傳來的 session 或直接傳入 user_id。

**方案：小馬直接用 Supabase client + service_role key 寫入。**
使用者的 RLS policy 仍保護一般前端存取，但小馬作為「授權助理」可透過 service_role bypass RLS。

**或者：改 Edge Function 加一個支援 user_id 的參數模式。**

## 分類自動判斷

1. 從關鍵字匹配：
   - 加油、柴油、油錢 → 燃料
   - 便當、午餐、早餐、晚餐、吃、餐廳、小吃 → 餐飲美食
   - 全聯、家樂福、超市 → 生活日用
   - 7-11、全家、超商 → 飲食或日用（依金額）
   - 檳榔、菸 → 香菸
   - 收入、薪資、月薪、工資 → 薪資

2. 找不到匹配 → 用預設 expense 分類

## 帳戶判斷

1. 從關鍵字匹配：
   - 現金、錢包 → 現金錢包
   - 玉山 → 玉山帳戶
   - 台新 → 台新帳戶
   - 永豐、大戶 → 永豐

2. 找不到 → 用預設 expense 帳戶

## 對話流程範例

```
小新: 加油 5000 現金
小馬: ✅ 已記錄！
      加油 TWD 5,000（燃料）從 現金錢包
      📊 本月已支出 TWD 6,420

小新: 收入 39760 月薪 玉山
小馬: ✅ 已記錄！
      月薪 TWD 39,760（薪資）入 玉山帳戶
      📊 本月收入 TWD 39,760

小新: 買便當 120
小馬: ✅ 已記錄！
      買便當 TWD 120（餐飲美食）
      📊 本月已支出 TWD 6,540

小新: 昨天跟小王吃熱炒 850 我付的
小馬: ✅ 已記錄！
      跟小王吃熱炒 TWD 850（餐飲美食）
      📊 本月已支出 TWD 7,390
```

## 需要的新功能

1. **快速記帳解析器**（parse 自然語言 → 結構化資料）
2. **分類/帳戶關鍵字對照表**（可設定）
3. **Edge Function 支援 user_id 直接寫入模式**
