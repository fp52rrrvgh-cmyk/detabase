# detabase — 功能差距與 OPC 設計 input

**產出日期**: 2026-06-01 | **用途**: 給 Codex 設計 OPC 架構的參考資料
**來源**: 小新舊 GAS+Sheets 系統（14+ 功能模組）vs 現有 detabase

---

## 一、小新的真實需求一句話總結

> 把所有財務資訊統整起來，利用 AI 告訴我未來方向、我該怎麼做才能最好。

這是決策增強系統，不是記帳 App。

---

## 二、完整功能模組對照

### 使用者背景（OPC 設計必須考慮）
- 26 噸大貨車司機，單人自用，不考慮多人協作
- 主控台：iPhone 15 Pro Max
- 使用場景：車上、夜車、搬貨現場、疲勞狀態、單手操作
- 核心需求：超低操作步數、行動優先、可中斷接續、高容錯

### 🟢 模組 A — 儀表板（戰情總覽）

| 功能 | GAS | detabase | 差距 |

## 三、對 OPC 設計的建議

### 建議的 Profile 分工

考量小新是單人使用者 + Codex 在主工程師角色，OPC 可以簡化：

```
小新 (PO + User)
  └── 決定需求、驗收、真實使用
  │
小馬 (參謀 + 情報 + 營運)
  └── 需求分析、優先級判斷、跨 session 記憶
  └── 將小新的需求轉譯為 spec
  └── 營運監控、異常回報
  │
Codex (主工程師 + 架構師 + 審查)
  └── 收到 spec 後實作
  └── 架構設計與技術決策
  └── 審查成果品質
```

### 建議的建置優先順序（功能價值 × 技術難度）

| 優先 | 功能 | 原因 |
|:----:|:-----|:------|
| P0 | **決策面板** | 這是舊系統核心價值，也是 detabase 最大差距 |
| P0 | **債務管理模組** | 你負債 $609K，這是剛需 |
| P1 | **設定頁（零錢盒/初始金額）** | 交易流程的前置條件 |
| P1 | **待確認佇列** | 完整記帳流程的關鍵環節 |
| P2 | **月報/年報** | 決策面板的輸出形式 |
| P2 | **預算 AI 建議** | 決策面板的子功能 |
| P3 | **固定資產 + 折舊** | 長期功能，不急 |
| P3 | **現金流預測** | 需累積足夠交易資料才有意義 |

### 技術注意事項

1. **staging only** — 所有功能在 detabase-staging 開發驗證，無 production
2. **行動優先** — 每個功能必須在 iPhone Safari 上可用
3. **AI 決策不是一次性** — 小新的 AI 建議需要考慮時間維度（本月的建議 vs 下個月的實際結果 vs 建議的校正）
4. **零錢盒是特殊邏輯** — 每天整理零錢丟入，年末才存入銀行。這在 schema 上需要一個特殊 account type + 自動轉帳排程
