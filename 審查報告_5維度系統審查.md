## 1. 程式碼審驗 (Code Review) — 分數：7/10

### 發現的問題

#### 🔴 高優先 — 必須修復

| # | 問題 | 位置 | 說明 |
|---|------|------|------|
| CR-1 | **SQL injection 風險 (ILike)** | `apps/web/app/dashboard/search/page.tsx` L245 | `ilike("description", \`%${f.keyword.trim()}%\`)` — 關鍵字直接內插，使用者可輸入 `%` 或 `_` 進行 pattern 探測。雖非傳統注入，但可能導致 unexpected 行爲。 |
| CR-2 | **any 型別大量泛濫** | `useDashboard.ts` L230-241, `useQuickCapture.ts` L149-207, 多處 | 大量使用 `as any[]` 和 `(rows ?? []) as any[]` — 完全繞過 TypeScript 型別檢查。優點是快速原型，但失去型別安全。 |
| CR-3 | **Promise 未處理 catch** | `useDashboard.ts` L399 (`void load()`)、多個 `loadData()` 中的 `void` | `void load({ signal: controller.signal })` 沒有 catch，若 load 內 throw 會 silent fail。同樣問題出現在 `CategoriesTab`, `BudgetsTab`, `AccountsTab`, `RulesTab`, `SubscriptionsTab` 的 `useEffect` 中。 |
| CR-4 | **useEffect 內 async 未處理錯誤** | `useQuickCapture.ts` L141-156, L168-183 | `supabase.auth.getSession().then(...)` 鏈式調用無 catch 處理，若 `getSession` 失敗則狀態停滯在 "loading" |

#### 🟡 中優先 — 應改善

| # | 問題 | 位置 | 說明 |
|---|------|------|------|
| CR-5 | **useDashboard 的 dependencies array 問題** | `useDashboard.ts` L394 | `useCallback` 的 deps 包含 `isCurrentMonth`，而 `isCurrentMonth` 由 `year` 和 `month` 計算得出，不該在 dep 中。雖不會造成 bug，但違反 React 慣例。 |
| CR-6 | **access_token 直接暴露給前端** | 多處 `headers` | `authorization: \`Bearer ${currentSession.access_token}\`` — 此爲 Supabase 標準用法，但 token 在 localStorage 中裸存，無 httpOnly cookie 保護。屬 PWA 類應用的常見取捨。 |
| CR-7 | **loadData 多次同時執行** | `budgets/page.tsx` L158-160 | 使用 `AbortController` 但註解說「不會取消 Supabase 查詢」，實際 controller 未傳入查詢。可能造成 stale state 覆蓋。 |
| CR-8 | **Edge Function 錯誤代碼不一致** | `void-finance-activity` 與 `set-budget` | `void-finance-activity` 很多錯誤都回傳 `void_not_allowed` (401/403)，前端的 `safeVoidFailureMessage` 與之對應的 mapping 需要準確對齊。 |
| CR-9 | **recharts 無 error boundary** | `DailyTrendChart.tsx`, `ExpensePieChart.tsx`, `YearOverYearChart.tsx` | 如果 API 回傳空資料或格式異常，recharts 可能 throw，只靠最外層的 ErrorBoundary 保護。 |

#### 🟢 低優先 — 可觀察

| # | 問題 | 說明 |
|---|------|------|
| CR-10 | `any` 型別在 `tx` prop 中 (page.tsx L24) | `{ tx: any }` 應該用具體型別 |
| CR-11 | `lookupFailed` 在 log-finance-activity 中是 string literal 而非 union member | 型別定義 `SafeFailureCode` 缺少 `"reference_lookup_failed"` (但實際有使用這個值) |

### 建議修復優先順序 (依序)
1. CR-1 (SQL injection-like pattern)
2. CR-3, CR-4 (未處理的 Promise catch)
3. CR-2 (型別安全 — 逐步以具體型別替代 `any`)
4. CR-7 (重複執行保護)
